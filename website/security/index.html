Security model

Updates are executable code. Treat the pipeline accordingly.

SparkleReleaseKit keeps public configuration separate from signing secrets and verifies the artifact users will actually receive.

Trust layersSecretsCIReleaseReport

Five complementary trust layers

HTTPS

Protects feeds and downloads in transit.

Sparkle EdDSA

Authenticates the exact update archive to the installed app in every mode.

App code signing

Provides structural integrity with ad-hoc signing or an Apple identity with Developer ID.

Developer ID

Optionally binds direct-distribution code to an Apple-verified developer identity.

Notarization

Optionally adds Apple's automated scan and Gatekeeper ticket.

The free route uses HTTPS, Sparkle EdDSA, checksums, a manifest, and ad-hoc app signing. Developer ID and notarization strengthen the Apple trust chain but never replace Sparkle signing.

Private means private

SUPublicEDKey is public and belongs in the app. The private EdDSA key, `.p12` certificates, `.p8` keys, passwords, and tokens belong only in Keychain or protected CI secrets.

Never paste private signing material into a coding-agent prompt, issue, log, configuration file, or commit.

Read the key management and recovery guide.

Least-privilege automation

Validation jobs receive read-only repository access and no release secrets. Production signing should run in a protected environment, with reviewed actions pinned to commit SHAs and explicit permissions.

Verify the release, not just the source

Run tests, build Release, inspect architectures and code signing, create the final ZIP or DMG, verify the exact EdDSA signature, checksum, and manifest, then perform an older-to-newer test update. Developer ID releases additionally require notarization and staple validation.

Report a vulnerability privately

Use GitHub private vulnerability reporting. Remove private keys, user data, and proprietary project source before submitting evidence.
